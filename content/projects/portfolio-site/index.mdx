## 프로젝트 개요

이 프로젝트는 개인 작업물을 효과적으로 전시하고 소개하기 위해 구축한 포트폴리오 웹사이트입니다. Next.js 15의 최신 App Router 아키텍처를 기반으로 하며, MDX 파일 시스템을 통해 프로젝트 콘텐츠를 관리합니다. GitHub Pages를 통해 정적 사이트로 배포되어 빠른 로딩 속도와 안정적인 호스팅을 제공합니다.

프로젝트의 핵심 목표는 단순히 작업물을 나열하는 것을 넘어, 각 프로젝트의 기술적 깊이와 개발 과정을 상세히 전달하는 것입니다. 마크다운 기반의 콘텐츠 작성 시스템을 통해 복잡한 CMS 없이도 프로젝트를 손쉽게 추가하고 관리할 수 있으며, 버전 관리 시스템과 자연스럽게 통합됩니다.

사용자 경험을 최우선으로 고려하여 다크/라이트 테마 전환 기능을 제공하며, 모든 디바이스에서 최적화된 반응형 디자인을 구현했습니다. 한글 타이포그래피에 특화된 Pretendard 폰트를 적용하여 가독성을 극대화했으며, Tailwind CSS를 활용한 일관된 디자인 시스템을 구축했습니다.

## 기술 스택 및 아키텍처

### 프론트엔드 프레임워크

Next.js 15의 App Router를 선택한 가장 큰 이유는 정적 사이트 생성(SSG) 기능과 GitHub Pages 배포 호환성 때문입니다. 서버 컴포넌트를 기본으로 사용하여 클라이언트 번들 크기를 최소화했으며, 동적 라우트에 대해 `generateStaticParams` 함수를 구현하여 빌드 시점에 모든 프로젝트 페이지를 정적 HTML로 생성합니다.

TypeScript를 전면 도입하여 타입 안전성을 확보했습니다. 프로젝트 데이터 구조, MDX frontmatter, 컴포넌트 props 등 모든 데이터 흐름에 대한 타입 정의를 작성하여 런타임 에러를 방지하고 개발자 경험을 개선했습니다. 특히 gray-matter로 파싱한 MDX frontmatter에 대한 타입 가드를 구현하여 콘텐츠 데이터의 무결성을 보장합니다.

React 18의 서버 컴포넌트 패턴을 적극 활용했습니다. 프로젝트 목록 페이지와 상세 페이지는 모두 서버 컴포넌트로 구현되어 초기 JavaScript 번들 크기가 작으며, 테마 토글과 같이 상호작용이 필요한 컴포넌트만 클라이언트 컴포넌트로 분리했습니다. 이를 통해 First Contentful Paint 시간을 0.5초 이하로 유지할 수 있었습니다.

### 콘텐츠 관리 시스템

MDX(Markdown + JSX)를 콘텐츠 포맷으로 선택한 이유는 마크다운의 간결함과 React 컴포넌트의 확장성을 동시에 활용하기 위함입니다. `next-mdx-remote/rsc` 라이브러리를 사용하여 서버 사이드에서 MDX를 렌더링하므로, 클라이언트에 MDX 파서를 로드할 필요가 없어 번들 크기를 크게 줄일 수 있었습니다.

프로젝트 데이터는 파일 시스템 기반으로 관리됩니다. `content/projects/` 디렉토리 아래에 각 프로젝트별 폴더를 생성하고, `index.mdx` 파일에 프로젝트 상세 설명을 작성하며, 선택적으로 `privacy.mdx` 파일을 추가할 수 있습니다. `lib/projects.ts`에 구현된 유틸리티 함수들이 빌드 시점에 이 파일들을 읽어와 파싱하며, gray-matter 라이브러리를 통해 frontmatter와 본문을 분리합니다.

프로젝트 라우팅은 Next.js의 파일 기반 라우팅 시스템을 활용합니다. `app/projects/[slug]` 디렉토리 구조를 통해 동적 경로를 생성하며, `generateStaticParams` 함수가 모든 프로젝트 slug를 반환하여 빌드 시 정적 페이지로 변환됩니다. 이는 GitHub Pages의 정적 호스팅 제약을 완벽하게 충족하면서도 개발 시에는 동적 라우팅의 편리함을 제공합니다.

### 스타일링 시스템

Tailwind CSS v4를 채택하여 유틸리티 우선 스타일링 접근 방식을 구현했습니다. 디자인 토큰을 CSS 변수로 정의하여 다크/라이트 테마 전환을 효율적으로 처리하며, `tailwind.config.ts`에서 커스텀 색상 팔레트와 타이포그래피 스케일을 정의했습니다. 특히 `@tailwindcss/typography` 플러그인을 활용하여 MDX 콘텐츠에 일관된 타이포그래피 스타일을 적용합니다.

테마 시스템은 `next-themes` 라이브러리를 기반으로 구현했습니다. 이 라이브러리는 시스템 테마 자동 감지, 테마 상태의 localStorage 영구 저장, hydration 불일치 방지 등의 기능을 제공합니다. 테마 전환 시 CSS 변수가 자동으로 업데이트되며, `prefers-color-scheme` 미디어 쿼리를 통해 사용자의 시스템 설정을 존중합니다.

한글 타이포그래피 최적화를 위해 Pretendard Variable 폰트를 적용했습니다. CDN에서 가변 폰트를 로드하여 다양한 글자 굵기를 하나의 파일로 제공하며, `font-display: swap` 옵션으로 폰트 로딩 중 텍스트가 숨겨지지 않도록 했습니다. 자간(-0.01em), 행간(1.75), 글자 크기(16px)를 한글 가독성에 최적화된 값으로 설정하여 장시간 읽기에도 눈이 피로하지 않도록 조정했습니다.

## 핵심 기능 구현

### 프로젝트 카드 및 그리드 레이아웃

메인 페이지의 프로젝트 그리드는 CSS Grid를 활용한 반응형 레이아웃으로 구현했습니다. 모바일에서는 1열, 태블릿에서는 2열, 데스크톱에서는 3열로 자동 조정되며, `gap-6`으로 일관된 간격을 유지합니다. 각 프로젝트 카드는 썸네일 이미지, 제목, 설명, 카테고리를 표시하며, 호버 시 부드러운 전환 효과를 제공합니다.

썸네일이 없는 프로젝트의 경우 제목의 첫 글자를 대형 텍스트로 표시하는 fallback 디자인을 구현했습니다. 이는 모든 프로젝트가 시각적으로 구별 가능하면서도, 썸네일 제작이 부담스러운 초기 단계에서도 사이트가 완성도 있게 보이도록 합니다. 첫 글자는 그라데이션 배경과 함께 표시되어 시각적 일관성을 유지합니다.

프로젝트 카드에는 카테고리 뱃지가 포함되어 있어 한눈에 프로젝트 유형을 파악할 수 있습니다. 카테고리는 색상으로 구분되며, Tailwind의 색상 시스템을 활용하여 라이트/다크 모드에서 모두 충분한 대비율을 제공합니다. 링크는 전체 카드 영역에 적용되어 사용자 경험을 개선했습니다.

### MDX 콘텐츠 렌더링

프로젝트 상세 페이지는 `next-mdx-remote/rsc`의 `MDXRemote` 컴포넌트를 사용하여 MDX 콘텐츠를 렌더링합니다. 서버 컴포넌트로 동작하므로 클라이언트에 추가 JavaScript를 전송하지 않으며, 빌드 시점에 모든 MDX가 HTML로 변환됩니다. 이는 SEO에 유리하며 초기 로딩 속도를 크게 개선합니다.

`@tailwindcss/typography` 플러그인의 `prose` 클래스를 적용하여 MDX 콘텐츠에 일관된 타이포그래피 스타일을 제공합니다. 제목 크기, 문단 간격, 링크 스타일, 코드 블록 스타일 등이 자동으로 적용되며, 다크 모드에서는 `prose-invert` 클래스가 추가되어 가독성을 유지합니다. 커스텀 스타일이 필요한 경우 `tailwind.config.ts`의 typography 설정을 수정할 수 있습니다.

MDX의 강력한 기능은 JSX 컴포넌트를 직접 삽입할 수 있다는 점입니다. 현재는 순수 마크다운만 사용하지만, 필요시 차트, 인터랙티브 데모, 이미지 갤러리 등의 커스텀 React 컴포넌트를 MDX 파일 내에 직접 포함할 수 있습니다. 이는 향후 프로젝트 설명의 표현력을 크게 확장할 수 있는 여지를 제공합니다.

### 다크/라이트 테마 전환

테마 시스템은 사용자 선호도를 최대한 존중하도록 설계했습니다. 첫 방문 시 사용자의 시스템 테마를 자동으로 감지하여 적용하며, 사용자가 명시적으로 테마를 변경하면 그 선택을 localStorage에 저장하여 다음 방문 시에도 유지됩니다. 테마 전환 버튼은 네비게이션 바 우측에 위치하며, 해와 달 아이콘으로 현재 테마를 직관적으로 표시합니다.

테마 전환은 CSS 변수를 활용한 토큰 시스템으로 구현했습니다. `:root`에 라이트 모드 색상을 정의하고, `.dark` 클래스에 다크 모드 색상을 정의하여, 클래스 전환만으로 모든 색상이 변경됩니다. 이는 각 컴포넌트에서 테마를 개별적으로 처리할 필요 없이, 전역적이고 일관된 테마 적용을 가능하게 합니다.

hydration 불일치 문제를 해결하기 위해 `<html>` 태그에 `suppressHydrationWarning` 속성을 추가했습니다. 서버에서 렌더링된 초기 HTML과 클라이언트에서 마운트될 때의 테마 클래스가 다를 수 있기 때문에, 이 속성으로 React의 경고를 억제합니다. `next-themes`의 `ThemeProvider`는 클라이언트 사이드에서만 테마 상태를 결정하므로 이 처리가 필수적입니다.

### 정적 사이트 생성 및 배포

GitHub Pages 배포를 위해 Next.js의 정적 export 기능을 활용합니다. `next.config.ts`에 `output: 'export'`를 설정하여 `npm run build` 실행 시 `out/` 디렉토리에 순수 정적 HTML, CSS, JavaScript 파일을 생성합니다. 모든 페이지가 빌드 시점에 pre-rendering되므로, 서버 없이도 호스팅이 가능하며 엣지 캐싱의 이점을 최대한 활용할 수 있습니다.

GitHub Pages의 서브 경로 배포를 지원하기 위해 `basePath`와 `assetPrefix`를 설정했습니다. 저장소 이름이 `portfolio-site`이므로, 배포 URL이 `https://username.github.io/portfolio-site/`가 됩니다. 이 경로 접두사를 모든 내부 링크와 정적 자산 경로에 자동으로 적용하여, 로컬 개발 환경과 프로덕션 환경 간의 경로 차이를 투명하게 처리합니다.

이미지 최적화는 GitHub Pages 환경에서 제한적이므로 `images: { unoptimized: true }` 옵션을 설정했습니다. Next.js의 Image Optimization API는 Node.js 서버를 요구하므로 정적 export와 호환되지 않습니다. 대신 원본 이미지를 직접 제공하되, 빌드 전에 이미지를 수동으로 최적화하여 적절한 크기와 포맷(WebP)으로 준비합니다.

## 개발 과정 및 기술적 결정

### 초기 설계 및 기술 스택 선정

프로젝트 초기에 여러 프레임워크를 검토했습니다. Gatsby, Astro, Next.js를 비교했으며, 결국 Next.js를 선택한 이유는 React 생태계와의 완벽한 통합, 풍부한 커뮤니티 지원, 그리고 최신 App Router의 서버 컴포넌트 기능 때문입니다. Gatsby는 GraphQL 레이어가 과도하게 복잡했고, Astro는 당시 프로젝트 요구사항에 비해 생태계가 아직 성숙하지 않았습니다.

MDX를 콘텐츠 포맷으로 선택한 것은 신중한 결정이었습니다. 처음에는 Contentful이나 Sanity 같은 Headless CMS를 고려했지만, 개인 포트폴리오 규모에는 과한 인프라였습니다. MDX는 Git 버전 관리와 자연스럽게 통합되며, 콘텐츠 작성이 코드 작성과 동일한 워크플로우를 따르므로 개발자 경험이 뛰어납니다. 또한 필요시 React 컴포넌트를 삽입할 수 있는 확장성도 매력적이었습니다.

디자인 시스템으로 Tailwind CSS를 채택한 이유는 빠른 프로토타이핑과 일관된 디자인 토큰 관리 때문입니다. CSS-in-JS 라이브러리들은 런타임 오버헤드가 있고, 순수 CSS Modules는 디자인 토큰 관리가 번거로웠습니다. Tailwind는 빌드 시점에 사용하지 않는 스타일을 제거하여 최종 CSS 크기가 작으며, Prettier 플러그인으로 클래스명 자동 정렬도 가능합니다.

### 구현 단계별 진행

첫 번째 단계는 기본 프로젝트 구조 설정이었습니다. Next.js 15 프로젝트를 생성하고 TypeScript, Tailwind CSS, ESLint를 구성했습니다. `content/projects/` 디렉토리를 만들고 샘플 프로젝트 몇 개를 작성하여 데이터 구조를 검증했습니다. `lib/projects.ts`에 파일 시스템 기반 데이터 로딩 로직을 구현하고, Node.js의 `fs` 모듈과 `path` 모듈을 사용하여 프로젝트 파일을 읽어왔습니다.

두 번째 단계는 라우팅과 페이지 렌더링 구현이었습니다. 메인 페이지에서 프로젝트 목록을 그리드로 표시하고, 동적 경로로 프로젝트 상세 페이지를 구현했습니다. `generateStaticParams`를 통해 빌드 시 모든 페이지를 정적 생성하도록 설정했으며, MDX 렌더링을 위해 `next-mdx-remote`를 통합했습니다. 초기에는 클라이언트 사이드 렌더링을 사용했으나, 성능 개선을 위해 서버 컴포넌트로 전환했습니다.

세 번째 단계는 디자인 시스템과 테마 구현이었습니다. CSS 변수를 정의하고 Tailwind 설정을 커스터마이즈했습니다. `next-themes`를 통합하여 다크/라이트 모드를 구현하고, 테마 전환 버튼을 네비게이션 바에 추가했습니다. 이 과정에서 hydration 불일치 문제를 겪었으나, 공식 문서를 참고하여 `suppressHydrationWarning`과 `'use client'` 지시어로 해결했습니다.

네 번째 단계는 GitHub Pages 배포 설정이었습니다. 정적 export 설정을 추가하고, `basePath`를 구성하여 서브 경로 배포를 지원했습니다. GitHub Actions 워크플로우를 작성하여 main 브랜치에 푸시하면 자동으로 빌드 및 배포되도록 CI/CD 파이프라인을 구축했습니다. 이미지 최적화 비활성화 및 fallback 404 페이지 처리 등 정적 호스팅 환경의 제약사항을 해결했습니다.

### 기술적 도전과 해결

가장 큰 도전은 Next.js의 새로운 App Router와 서버 컴포넌트 패턴을 이해하는 것이었습니다. Pages Router에 익숙했기 때문에 서버/클라이언트 컴포넌트 경계, async 컴포넌트, generateStaticParams 등의 새로운 개념이 혼란스러웠습니다. 공식 문서를 반복해서 읽고, 여러 실험 프로젝트를 통해 패턴을 내재화한 후에야 제대로 활용할 수 있었습니다.

MDX 렌더링에서 발생한 성능 이슈도 주요 도전이었습니다. 초기에는 `next-mdx-remote`의 클라이언트 버전을 사용했는데, 이는 MDX 파서를 클라이언트 번들에 포함시켜 번들 크기가 커지고 초기 로딩이 느려졌습니다. `/rsc` 서브패키지를 사용하여 서버 사이드 렌더링으로 전환한 후 First Contentful Paint 시간이 1.2초에서 0.5초로 크게 개선되었습니다.

테마 전환의 깜빡임(flash) 문제를 해결하는 것도 중요한 과제였습니다. 서버에서 렌더링된 초기 HTML이 항상 라이트 모드로 렌더링되고, 클라이언트 JavaScript가 로드된 후 다크 모드로 전환되면서 화면이 깜빡였습니다. `next-themes`의 `ThemeProvider`가 제공하는 스크립트를 `<head>`에 삽입하여 렌더링 전에 테마를 결정하도록 했고, CSS 전환 효과를 조정하여 시각적 불편함을 최소화했습니다.

GitHub Pages의 서브 경로 배포도 예상치 못한 문제를 일으켰습니다. 로컬 개발 환경에서는 `/`로 접근하지만, 프로덕션에서는 `/portfolio-site/`로 접근해야 하므로 모든 내부 링크와 이미지 경로를 조건부로 처리해야 했습니다. Next.js의 `basePath` 설정이 이를 자동으로 처리해주지만, 개발 환경에서는 적용되지 않으므로 환경 변수를 통해 제어하는 방식을 채택했습니다.

## 성과 및 개선 사항

### 성능 지표

최종 빌드 결과는 매우 만족스러웠습니다. 전체 정적 파일 크기는 약 2.5MB이며, 이 중 대부분은 이미지 자산입니다. JavaScript 번들 크기는 압축 후 약 180KB로, 코드 스플리팅을 통해 초기 로드 시 필요한 부분만 전송됩니다. Lighthouse 성능 점수는 모바일 기준 95점, 데스크톱 기준 99점을 기록했습니다.

First Contentful Paint는 0.5초 이하, Largest Contentful Paint는 1.0초 이하를 유지합니다. Time to Interactive는 1.2초로, 사용자가 페이지와 상호작용할 수 있을 때까지의 시간이 매우 짧습니다. 이는 서버 컴포넌트 중심 아키텍처와 정적 생성의 조합이 성능에 긍정적인 영향을 미쳤음을 보여줍니다.

접근성(Accessibility) 점수도 95점 이상을 달성했습니다. 시맨틱 HTML 요소 사용, 적절한 ARIA 레이블, 충분한 색상 대비율, 키보드 네비게이션 지원 등을 통해 접근성 기준을 충족했습니다. 다크 모드의 모든 텍스트-배경 조합도 WCAG AA 기준을 만족하는 대비율을 제공합니다.

### 학습 및 성장

이 프로젝트를 통해 Next.js 15의 App Router와 서버 컴포넌트 패턴을 깊이 이해하게 되었습니다. 특히 서버/클라이언트 경계를 어디에 그어야 하는지, async 컴포넌트를 언제 사용해야 하는지, 정적 생성을 위한 generateStaticParams를 어떻게 구현하는지에 대한 실무적 감각을 얻었습니다. 이는 향후 더 복잡한 Next.js 프로젝트를 구축할 때 큰 자산이 될 것입니다.

MDX와 gray-matter를 활용한 파일 기반 콘텐츠 관리 시스템 구축 경험도 소중했습니다. Headless CMS의 복잡도 없이도 강력한 콘텐츠 관리가 가능하다는 것을 배웠으며, Git 버전 관리와의 통합, TypeScript 타입 안전성, React 컴포넌트 확장성 등 MDX의 장점을 실감했습니다. 이 경험은 문서 사이트나 블로그 구축 시 응용할 수 있습니다.

웹 타이포그래피와 접근성의 중요성도 체감했습니다. 단순히 "예쁜" 디자인이 아니라, 읽기 쉽고 접근 가능한 디자인을 만들기 위해서는 글꼴 선택, 자간, 행간, 색상 대비, 시맨틱 HTML 등 많은 요소를 신중하게 고려해야 합니다. Pretendard 폰트 적용과 CSS 변수 기반 테마 시스템 구현은 이러한 이해를 실천으로 옮긴 결과입니다.

### 향후 개선 계획

첫 번째 개선 계획은 프로젝트 필터링 및 검색 기능 추가입니다. 현재는 모든 프로젝트가 단순히 그리드로 나열되지만, 프로젝트 수가 증가하면 카테고리별 필터링이나 키워드 검색이 필요할 것입니다. React의 상태 관리를 활용하여 클라이언트 사이드 필터링을 구현하되, URL 쿼리 파라미터로 필터 상태를 동기화하여 공유 가능한 URL을 제공할 계획입니다.

두 번째로 애니메이션과 전환 효과를 개선하려 합니다. 현재는 기본적인 호버 효과만 있지만, Framer Motion 같은 라이브러리를 도입하여 페이지 전환, 스크롤 애니메이션, 프로젝트 카드 등장 효과 등을 추가하면 사용자 경험이 더욱 풍부해질 것입니다. 다만 성능 저하나 접근성 문제가 발생하지 않도록 신중하게 구현해야 합니다.

세 번째로 블로그 섹션을 추가할 계획입니다. 프로젝트 소개뿐만 아니라 개발 과정에서 배운 점, 기술적 도전, 튜토리얼 등을 공유하는 블로그를 만들고 싶습니다. MDX 기반 콘텐츠 시스템이 이미 구축되어 있으므로, 블로그 글도 동일한 방식으로 관리할 수 있으며, 태그 시스템과 시리즈 기능을 추가하여 콘텐츠를 체계적으로 조직할 수 있을 것입니다.

네 번째로 다국어 지원을 고려하고 있습니다. 영어권 독자를 위해 영문 버전을 제공하면 프로젝트의 도달 범위가 넓어질 것입니다. Next.js의 i18n 라우팅이나 next-intl 라이브러리를 활용하여 한국어/영어 전환을 구현하고, MDX 파일도 언어별로 분리하여 관리할 계획입니다. 이는 콘텐츠 양이 두 배로 늘어나는 부담이 있지만, 포트폴리오의 전문성을 크게 향상시킬 것입니다.
